**Existing Patterns (from codebase):**
- Models in `meal_planner_app/models.py` use UUID pk, TextChoices for enums (see MealType)
- Views use LoginRequiredMixin + CreateView/UpdateView pattern
- Templates use `.input-dark`, `.btn-pill` classes with DaisyUI dark theme
- Forms in `meal_planner_app/forms.py` receive `request` via get_form_kwargs
- Success redirects use reverse_lazy with namespace: `meal_planner:planner`
- Household scoping: `self.request.user.household`

**Key Design Decisions:**
- D-01: MealPreferences as one-to-one with Household (not User) — preferences are shared per household, same scoping as MealPlan/Recipe
- D-02: Cuisines stored as ArrayField of CharField (Postgres native) — no separate cuisine model needed for v1.2
- D-03: Dietary restrictions stored as ArrayField of CharField — same approach as cuisines
- D-04: Excluded ingredients stored as ArrayField of CharField — simple text tags
- D-05: Cooking effort stored as CharField with TextChoices (quick/moderate/elaborate)
- D-06: Servings stored as PositiveSmallIntegerField (default 2, min 1, max 8)
- D-07: Single UpdateView for editing preferences (get_or_create pattern for rendering)
- D-08: Model lives in meal_planner_app (closest to planning domain)

**Reference Data for Cuisines:**
Italian, Mexican, Asian, American, Mediterranean, Indian, Japanese, Thai, Greek, French, Korean, Middle Eastern, Southern, Caribbean, Cajun, German, Vietnamese, Brazilian, Spanish, Moroccan

**Reference Data for Dietary Restrictions:**
Vegetarian, Vegan, Gluten-Free, Dairy-Free, Low-Carb, Keto, Nut-Free, Egg-Free, Soy-Free, Pescatarian
